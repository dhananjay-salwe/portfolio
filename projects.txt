

1. Project: F-Widget / face-widget-dashboard
A multi-tenant Laravel 11 SaaS platform that lets dashboard users create embeddable face recognition widgets for their own websites.

Widget management (CRUD)
Each widget has: name, mode (floating/embedded), position, primary color, button color, welcome title, welcome message, button text, request limit, allowed domains, allowed pages, active status, and a show_start_button toggle (manual click vs auto-start camera).
Two embed modes
Floating mode injects a fixed-position card anywhere on the page. Embedded mode requires a <div id="face-widget-container"> placement div and injects inline. Both use Shadow DOM isolation so the widget CSS never leaks into or from the host page.
Domain security (3-layer)
First, WidgetController::script() checks the request's Origin/Referer against the widget's allowed domains list. Second, initWidget() issues a short-lived HMAC token (10-minute TTL, signed with APP_KEY) that the widget must attach to every track-hit request. Third, DomainController::findVerifiedDomain() checks that the domain has been ownership-verified before allowing the widget to fully initialise on production.
Domain verification
Dashboard users add their domains and can verify ownership three ways: meta tag verification (checks a custom URL or homepage for <meta name="face-widget-verify" content="token">), DNS TXT record verification, or Force Verify (dev bypass that marks it verified instantly without HTTP checks).
Usage tracking (success-only)
api_hits is incremented only when result.final === true && result.face_id — meaning a real face was recognised. Page loads, refreshes, failed scans, and camera errors never count. Rate limited at 10 requests per minute per widget_id + session_id to prevent replay attacks.
Session persistence
Each widget generates a SESSION_ID via crypto.randomUUID() stored in localStorage under key fw_session_{widget_id}. This persists across page refreshes so the same browser session doesn't generate duplicate hits.
Camera start modes
show_start_button (set per widget in the dashboard) controls whether the widget shows a clickable Start button or auto-starts the camera 1.5 seconds after load. The live preview in the create/edit form reflects this in real time — the button dims and strikethroughs in auto-start mode.
Reset API
window.FaceWidget.reset() is exposed as a global on the host page. Calling it stops scanning, stops the camera, resets all state and the usage flag, then either re-enables the button (button mode) or restarts the camera after 800ms (auto-start mode). Customers copy the reset snippet from the widget's show page.
Face API integration
The external EC2 face recognition API URL is configured via FACE_API_URL in .env and injected into serve.blade.php through WidgetController::serveScript(). Widget calls init_register() then sends base64 JPEG frames to process_frame() at 800ms intervals. Results include status, final, face_id, and stop_sending flags.
Recognition result delivery
On successful recognition the widget fires both window.dispatchEvent(new CustomEvent('faceWidgetRecognized', ...)) and window.postMessage({ type: 'FACE_RECOGNITION_COMPLETE', faceId }) so host pages can listen either way and act on the face ID (e.g. fill a hidden form field).
Local/dev bypass
Requests from localhost, 127.x.x.x, .test, .local, and file:// origins automatically skip all domain verification and token checks so development works without registering domains.
Docker production setup
12 files generated for containerised deployment: Nginx config, PHP-FPM, docker-compose.yml, Dockerfile, environment handling, and health checks.

2. Project Overview
Project Name: Swayamsiddh Official Website

Live Domain: https://swayamsiddh.in

Organization: Swayamsiddh – Parental Organisation for Intellectually Disabled Children (Chatrapati Sambhajinagar, Maharashtra).

Developed By: aPLS Web Developementelopment

Primary Objective: To provide a modern, accessible, and easily manageable digital platform for the NGO to share their mission, facilities, and latest news with the local community and potential supporters.

Technology Stack
Backend Framework: Laravel (PHP 8.3)

Frontend Technologies: Blade Templating, HTML5, CSS3, JavaScript

UI Framework: Bootstrap 5

Animations: AOS (Animate On Scroll) Library

Database: MySQL

Authentication: Customized Laravel Breeze

Public-Facing Features (Frontend)
The user interface is designed with a focus on accessibility and community engagement.

Bilingual Architecture: A seamless language switcher allows users to toggle the entire website interface and content between English and Marathi, storing the preference in the session.

Accessibility Tools: A custom-built, multi-step font resizer (A-, A, A+) allows visually impaired users to scale the text from 70% to 140%. The preference is saved via localStorage so it persists across page loads.

Dynamic News & Events: A dedicated news portal that automatically categorizes posts into "Current Year" and "Past Years."

Responsive UI: Mobile-first design implementation, including a responsive fixed-top navbar and a non-intrusive cookie consent banner engineered to fit all device viewports.

Core Pages: Home, About Us, Facilities, Gallery, and Contact.

Secure Admin Dashboard (Backend)
The backend is highly locked down to protect the organization's data while providing easy content management tools for the staff.

Secret Portal Entry: Standard login routes are disabled. Administrators must access the dashboard through a custom, hidden URL (/swayam-portal) to mitigate automated brute-force attacks.

News & Event Manager: Full CRUD (Create, Read, Update, Delete) capabilities. Admins can upload photos/videos, write titles and content in both English and Marathi, and even rotate images directly from the dashboard.

Inquiry Management: A dedicated view to read and manage messages submitted through the public Contact form.

Dynamic Contact Details: A module allowing admins to instantly update the school's address, phone numbers, and email on the public-facing site without touching the code.

User Management: Master admins can add or remove secondary admin accounts to control who has access to the dashboard.

Server & Deployment Architecture
The application is optimized and deployed for a live shared hosting environment.

Hosting Provider: Hostinger

Storage Configuration: Due to shared hosting restrictions blocking traditional symlinks (php artisan storage:link), the filesystem is custom-configured. The public disk maps directly to public/storage, allowing image uploads to bypass security blocks and render instantly on the live site.

Environment: Production mode enabled (APP_ENV=production) with strict error masking to protect backend logic from public view.

Mail Routing: Connected via SMTP to handle automated emails and contact form submissions securely.


🛠️ Tools & Technologies Used
Frontend (Client-Side)

HTML5 & CSS3: For semantic structure and custom styling.

JavaScript (Vanilla): For DOM manipulation, custom accessibility tools (font resizer), and dynamic UI interactions.

Bootstrap 5: For a highly responsive, mobile-first layout and pre-built grid system.

Blade Templating Engine: Laravel’s native templating engine for dynamic content rendering and layout inheritance.

AOS (Animate On Scroll): Lightweight library for smooth scroll animations.

MEDIAPIPE FACE DETECTION:
- Real-time face detection model
- Confidence: 0.45 (40% tolerance for harsh lighting)
- Fast inference on CPU/GPU
- Returns: Face bounding box, confidence score
- Used for: Initial face detection

MEDIAPIPE FACE MESH:
- 468 3D facial landmarks
- Tracks eyes, nose, mouth, face contours
- Used for: Liveness detection, movement tracking
- Confidence: 0.45
- Indexes: Eye (33, 133, 362, 263), Nose (1), Mouth (13, etc.)

ARCFACE (Face Embedding):
- ONNX model (face_embedder.onnx)
- Input: 112x112 RGB image (aligned face)
- Output: 512-dimensional vector (embedding)
- Purpose: Face matching and identification
- Distance metric: Cosine similarity
- Threshold: 0.4 (duplicate detection)

MINIFASNET V2 (Spoof Detection):
- ONNX model (MiniFASNetV2.onnx)
- Input: 80x80 RGB image
- Output: Probability [Spoof, Live]
- Purpose: Detect fake videos, photos, screens
- AI Threshold: < 0.20 is spoof (being adjusted to 0.35)
- Confidence needed: 0.85+ for high confidence

CLAHE (Brightness Normalization):
- Algorithm: Contrast Limited Adaptive Histogram Equalization
- Parameters: clipLimit=2.0, tileGridSize=(8,8)
- Purpose: Handle harsh/bright lighting
- LAB Color Space: Process L (lightness) channel
- Result: Normalized brightness, preserved features

ONNX RUNTIME:
- Cross-platform ML inference
- Fast execution on CPU
- No GPU required
- Supports multiple backends
- Thread-safe inference

================================================================================
 ANTI-SPOOFING & LIVENESS DETECTION
================================================================================

LIVENESS DETECTION METHODS:

Method 1: Biological Motion Detection
  - Tracks breathing patterns
  - Eye movement analysis
  - Natural head movement
  - Tremor/micro-movements detection

Method 2: Micro-Movement Analysis
  - Detects subtle facial movements
  - Breathing detection (nose movement)
  - Eye saccades
  - Facial tremor
  - Consistency scoring

Method 3: Blink Detection
  - Eye aspect ratio tracking
  - Blink counting
  - Natural blink timing
  - Minimum 1 blink for acceptance

Method 4: 3D Geometry Analysis
  - Parallax detection (eye position change)
  - Yaw estimation (head rotation)
  - Fisheye ratio (facial distortion)
  - 3D depth estimation from 2D

Method 5: AI-Based Detection (MiniFASNet)
  - Neural network classification
  - Detects unnatural lighting patterns
  - Identifies flat surfaces (photos, screens)
  - Confidence scoring
  - Averaging over 12 frames

SPOOF DETECTION METHODS:

Spoof Type 1: Printed Photos
  - Glossiness detection
  - Printer frequency analysis
  - Paper texture detection
  - Static edges detection
  - Confidence: Score > 58, 3+ indicators
  - AI Detection: 0.85+ confidence required

Spoof Type 2: Screen/Display
  - Screen glare detection
  - Moire pattern detection
  - Pixel grid detection
  - Display flicker detection
  - Banding detection
  - Confidence: Score > 55, 2+ indicators

Spoof Type 3: Video Replay
  - Compression detection (blockiness)
  - Smooth motion detection (unnatural)
  - Periodic motion detection (loops)
  - Frame repetition detection
  - Loop detection (13+ frame repeats)
  - Confidence: Score > 55, 2+ indicators

Spoof Type 4: Surface Analysis
  - Rigidity detection (flat objects)
  - Planarity detection (2D vs 3D)
  - Texture static detection
  - Translation consistency
  - Zoom detection (suspicious scaling)

SCORING SYSTEM:

LivenessScorer:
  - Positive Points: +10 to +40 for live evidence
  - Negative Points: -150 to -25 for spoof evidence
  - Final Score: 0-100 range
  - ACCEPT: Score >= 55 AND live evidence present
  - REVIEW: Score >= 35 (borderline cases)
  - REJECT: Score < 35 OR strong spoof evidence

Decisions:
  - ACCEPT: Confidence >= 55, biological motion present
  - REVIEW: Needs more analysis (35-54 score)
  - REJECT: Not enough evidence or spoof detected

  - UPDATE: Update face data
  - DELETE: Remove face record

ISSUE 4: 502 Bad Gateway
  Symptom: Nginx returns 502 error
  Causes:
    - Gunicorn not running
    - Port 5000 not accessible
    - Timeout (processing > 120s)
  Solution:
    - Check Gunicorn: sudo systemctl status face-recognition
    - Check port: sudo lsof -i :5000
    - Increase timeout in Nginx config
    - Restart: sudo systemctl restart face-recognition

ISSUE 5: Duplicate Not Detected
  Symptom: Same user registers twice with different face_id
  Root Cause: load_stored_faces() not called in init_register()
  Solution: Add function call in registration path
  Status: ✅ Fixed (FIX #4)

ISSUE 6: MySQL Connection Fails
  Symptom: "Unable to connect to MySQL"
  Causes:
    - MySQL not running
    - Wrong credentials
    - Cryptography missing
  Solution:
    - Start MySQL: sudo systemctl start mysql
    - Check credentials in db.py
    - Install cryptography
    - Check firewall (port 3306)

ISSUE 7: Zoom Detection False Positives
  Symptom: Normal head movements detected as zoom
  Root Cause: Thresholds too strict
  Solution: Increase zoom thresholds
  Status: ✅ Fixed (FIX #2)

ISSUE 8: Memory Leaks
  Symptom: Memory usage grows over time
  Causes:
    - Unbounded deques
    - Session not cleaned up
  Solution:
    - Use maxlen in deques (automatic rotation)
    - Clean up session after timeout
    - Monitor with: top, free -h
